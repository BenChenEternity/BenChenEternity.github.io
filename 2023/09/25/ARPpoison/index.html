Analyze the result

The ARP table of the target machine has been successfully poisoned. After poisoning, the MAC addresses in the ARP table have all been changed to Kali’s MAC address. This means that any access to any of these four addresses, 192.168.111.1, 192.168.111.2, 192.168.111.131, and 192.168.111.254, will be redirected to 192.168.111.131 on Kali.

Next, we will use Wireshark for packet capture and verification:

The first packet is an ARP packet, which may be the host’s attempt at verification (because ARP cache already has the information, theoretically, ARP messages shouldn’t be generated). However, it can be observed that the host resolved the MAC address for 192.168.111.1 as 00:0c:29:d0:99:19, which corresponds to Kali’s hardware address.

In reality, the packet’s destination address reached 192.168.111.131, not 192.168.111.1.

Kali (192.168.111.131) received an Echo Request (Ping) and responded with a Reply.

In real life, if ARP spoofing occurs, the network requests generated when users access a website will be intercepted by the attacker. The attacker can act as an intermediary and monitor the traffic by forwarding requests to the original address.

ARP spoofing is not easily detectable by ordinary users because they typically do not deliberately check or record ARP tables, and even if they do, they do not pay attention to MAC addresses.

ARP spoofing also has downsides for attackers:

  • If users refresh their ARP tables, ARP spoofing becomes ineffective.
  • If users restart their computers or use ARP refresh commands, the ARP table is refreshed.
    • Windows: arp -d
    • Linux: sudo ip -s -s neigh flush all

Verification is as follows:

Execute arp -d to refresh the ARP cache and check that the ARP cache is empty.

After packet capture, it is observed that this time the packets were correctly sent to 192.168.111.1.
